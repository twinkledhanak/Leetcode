Comparison of Spring Boot, Flask, and Django across key development concepts

At a glance

What each framework is, who uses it, and when to pick it.

Spring Boot Java · Enterprise

Full-stack, opinionated, annotation-driven. Built for large teams and enterprise systems. You already know this.

Used by: Banks, fintechs, large enterprise SaaS

Strengths: Dependency Injection (DI) container via Spring Context (you declare beans, the framework wires them together), huge and mature ecosystem (Spring Data, Spring Security, Spring Cloud), strong type safety and IDE tooling, very good fit for microservices and complex enterprise architectures.

Pain / trade-offs: verbose configuration and annotations once projects grow, higher memory usage and slower startup time compared with lightweight Python frameworks, steep learning curve around proxies, bean scopes, and auto-configuration magic.

How it runs / deploys: Option A (common now): run Spring Boot as a single app (JAR). It starts its own Tomcat internally.
Flow: website → load balancer → Spring Boot app → your controller method.

Option B (what your team did):

SPRINGBOOT BASED ARCHITECTURE BUT LEGACY TOMCAT DEPLOYMENT (USING EXTERNAL TOMCAT RUNNING ON UNIX SERVERS)

1. Software ArchitecturePattern: Microservices Architecture.
Implementation: The team successfully refactored a legacy monolithic application by breaking its business domains down into \(N\) small, self-contained, loosely coupled services (e.g., User Service, Payment Service).Framework: Every microservice component was built as an independent Spring Boot application.

2. Application PackagingBuild Artifact: The team configured the build automation tool (Maven or Gradle) to package each Spring Boot application into a WAR (Web Application Archive) file, rather than an executable JAR.Target Environment: Packaging as a WAR meant the code was intentionally designed to be a "tenant" dependent on a servlet container, rather than using Spring Boot's standalone capabilities.

3. Infrastructure and Deployment (The Core Complexity)Hardware Tier: The production environment consisted of 6 separate physical or virtual Unix operating system servers.Topology: The team executed a highly redundant "All-on-All" deployment strategy for high availability and fault tolerance. Every single microservice was deployed onto every single Unix server.Server Middleware: Instead of utilizing embedded runtimes, the Operations team manually installed and managed multiple independent Apache Tomcat instances (External Tomcat) on each individual Unix server.Resource Conflict Mitigation: Because multiple Tomcat instances lived on the same Unix operating system, the team had to manually externalize and reconfigure the network port allocations (e.g., 8080, 8081, 8082) inside each Tomcat instance's server.xml file to avoid network binding conflicts.Deployment Mechanism: Deployment required copying each distinct service's .war file directly into the corresponding Tomcat instance's local webapps/ directory on all 6 machines.

4. Networking and Traffic RoutingInbound Flow: A Load Balancer sat at the front gate to accept public traffic.Reverse Proxy Routing: The load balancer read incoming HTTP request paths (e.g., /api/users vs /api/payments) and used custom routing rules to map and forward that traffic to the correct server IP address and corresponding port number of the specific Tomcat instance running that service.

The Core Design: A WAR Cannot Run By Itself
A WAR file is completely dead on its own. If you double-click it or try to run it via the command line, nothing happens.
It does not contain a web server.
It is designed strictly as a plugin for a Servlet Container (like Apache Tomcat, WildFly, or GlassFish).
The container acts as the operating system for your WAR file.

Inside a WAR File: The Standard Structure
Because an external Tomcat server has to read WAR files from hundreds of different companies, every WAR file must follow an identical internal folder structure.
If you change a single folder name, Tomcat will reject it.
Here is what the inside of your team's user-service.war looked like:
📂 user-service.war (The root folder)
📄 index.html / login.jsp (Static frontend web pages, if your app had them)
📂 WEB-INF/ (The most critical, private folder. The outside internet cannot look inside here)
📄 web.xml (The deployment descriptor. The instruction manual that tells Tomcat how to route URLs to your Java code)
📂 classes/ (Your actual compiled Spring Boot Java code — the .class files)
📂 lib/ (All the third-party .jar libraries your code depends on to function)

How Tomcat Processes a WAR (Deployment)
When your operations team dropped user-service.war into Tomcat's webapps/ folder, a process called Deployment (or "Exploding") happened automatically:
Detection: Tomcat continuously watches its webapps/ folder. It detects the new file.
Extraction: Tomcat automatically unzips the WAR file into a normal folder of the same name.
Bootstrapping: Tomcat reads the WEB-INF/web.xml file to understand what your app needs.
Lifecycle Management: Tomcat boots up your Spring Boot application context, allocates memory to it, and opens up the network port to start feeding HTTP requests directly into your code.

In your previous team's setup
The Built-In Server Was Turned Off: Spring Boot's famous built-in, embedded Tomcat feature was completely deactivated or bypassed.
The External Server Ran the Show: Your Spring Boot code was stripped down into a passive WAR file and forced to run inside those heavy, standalone Tomcat programs installed manually on the Unix operating systems.
Essentially, your team used Spring Boot as a framework to write the microservice features, but they completely rejected Spring Boot's modern deployment method (Option A) in favor of the legacy Tomcat-as-a-landlord method (Option B). Lot of redundancy, it is a Springboot app with embedded tomcat but we were still using external tomcat server to deploy the same code.
With Option A, we just use the embedded tomcat server for each of the Springboot service. No external tomcat servers on Unix Servers. Every single Spring Boot service running in production uses its own separate, private Tomcat instance. how does concurrency work? It is still one Springboot server. Does scaling here would mean, having same springboot app present in multiple unix servers? Scaling absolutely means running copies of that exact same Spring Boot app across multiple Unix servers. Level 1: Internal Concurrency (Inside One Spring Boot App)Even though it is "one Spring Boot server," a single Tomcat instance is not limited to handling one request at a time. Tomcat is a multithreaded server.When a single user-service.jar boots up, its embedded Tomcat automatically creates a Thread Pool (usually 200 threads by default). If 50 users hit your User Service at the exact same millisecond, Tomcat assigns 50 different threads from its internal pool to handle them.All 50 users are processed concurrently inside that single running JAR process.


Level 2: External Scaling (When One JAR Isn't Enough)What happens if 5,000 users hit your User Service at once? The 200 internal threads will get maxed out, the CPU will spike, and the single app will slow down or crash.This is exactly where your 6 Unix servers and your Load Balancer come into play. To scale, you spin up identical copies of the user-service.jar across your servers: You run java -jar user-service.jar on all 6 Unix servers.You now have 6 separate processes, each running its own internal Tomcat, each with its own pool of 200 threads.The Load Balancer sits in front of all 6 servers. When traffic pours in, it acts like a traffic cop, spraying the requests evenly across all 6 servers (\(5,000 \div 6\)).

Flask Python · Micro

Micro-framework. Gives you a server and routing — you wire everything else yourself. Closest to raw Express.js.

Used by: ML teams, data science, small APIs, prototypes

Strengths: extremely small core and minimal abstractions (you can understand the entire request flow by reading a few files), very fast to spin up a proof-of-concept or thin API over an ML model, flexibility to pick your own stack for every concern (ORM, validation, auth, background jobs).

Pain / trade-offs: no built-in Dependency Injection container or opinionated architecture, you assemble everything — auth, ORM, validation, configuration, testing strategy — yourself, larger teams can easily end up with inconsistent patterns if you don’t enforce conventions.

How it runs / deploys: Local: flask run (dev server).
Prod: run Flask with a production server like Gunicorn/uWSGI/Waitress (often behind Nginx/load balancer).
Flow: website → load balancer → (Nginx optional) → Gunicorn/uWSGI → Flask route function.

The Practical Consequences
Embedded Tomcat Gunicorn
Concurrency model Threads inside 1 process Multiple processes
Default concurrency 200 threads (2 × CPU cores) + 1 workers
Memory One shared memory space Each worker has its own memory copy
Memory usage Lower (threads share memory) Higher (each process is a full copy)
A crash in one worker Can affect all threads (shared memory) Other workers unaffected (isolated processes)
Startup time One process starts N processes start (slightly slower)
Why this model Java has no GIL — threads truly parallel Python GIL makes threads useless for parallelism
The Memory Consequence — The Big Tradeoff
This is the real cost of Gunicorn's process model. 
Embedded Tomcat
───────────────
1 process using 512MB RAM
200 threads share that 512MB
Total RAM: 512MB

Gunicorn (4 workers)
────────────────────
Worker 1: 512MB RAM (full copy of Flask app)
Worker 2: 512MB RAM (full copy of Flask app)
Worker 3: 512MB RAM (full copy of Flask app)
Worker 4: 512MB RAM (full copy of Flask app)
Total RAM: 2048MB (4× more)

Each Gunicorn worker is a full independent copy of your Flask app loaded into memory. 4 workers = 4 copies of everything. This is why Python web servers need more RAM than equivalent Java servers handling the same traffic.

Bottom Line
Tomcat thread model works for Java because:
Java threads → truly parallel → no GIL → efficient

Gunicorn process model works for Python because:
Python threads → fake parallel due to GIL → useless
Python processes → each has own GIL → truly parallel → efficient
Tradeoff: more RAM per worker

The difference isn't a design preference. It's a language-level constraint forcing a different architecture. If Python didn't have the GIL, Gunicorn would just use threads like Tomcat does.
Django Python · Full-stack

Batteries-included. ORM, admin, auth, migrations all built-in. Closest philosophy to Spring Boot.

Used by: Content platforms, agencies, internal tools, established startups

Strengths: very fast CRUD and back-office development thanks to built-in admin UI, integrated stack (ORM, auth, sessions, templates, forms) that just works together, clear conventions around project/apps that help medium-sized teams stay organized.

Pain / trade-offs: opinionated “Django way” of doing things (URL routing, class-based views, ORM patterns), less natural fit for ultra-high-performance microservices compared to lighter frameworks, can feel restrictive when you want to deviate heavily from the standard stack.

How it runs / deploys: Local: python manage.py runserver (dev server).
Prod: run Django with Gunicorn/uWSGI (most common) behind Nginx/Apache/load balancer.
Flow: website → load balancer → Nginx/Apache → Gunicorn/uWSGI → Django view (or DRF view).

Mental model: Spring Boot and Django share the same philosophy — "framework does the heavy lifting, you configure." Flask is the opposite — "you build what you need." If you're comfortable with Spring Boot, Django will feel familiar. Flask will feel liberatingly simple but also naked.

Django

CLI generates folder structure. A "project" contains "apps" (modules). manage.py is your main tool for everything.

Key difference: Flask is one file. Django scaffolds a full folder structure. Spring Boot generates an entire Maven/Gradle project. Django's manage.py is your Swiss army knife — migrations, running server, creating superusers — like Spring Boot CLI but more central.

Routing

How URL paths map to your handler functions/methods.

Spring Boot
@RestController
@RequestMapping("/articles")
public class ArticleController {

  @GetMapping("/{id}")
  public Article get(
    @PathVariable Long id) {
    ...
  }

  @PostMapping("/")
  public Article create(...) {
    ...
  }
}

Routing lives ON the class and method via annotations. Routes are co-located with logic.

Flask
# Routes via decorator
# directly on functions

@app.route('/articles/<int:id>',
           methods=['GET'])
def get_article(id):
    ...

@app.route('/articles/',
           methods=['POST'])
def create_article():
    ...

Decorator on each function. Simple and explicit. All in one file or split with Blueprints.

Django
# urls.py (separate file)
urlpatterns = [
  path(
    'articles/<int:id>/',
    views.ArticleView.as_view()
  ),
  path(
    'articles/',
    views.ArticleListView.as_view()
  ),
]

Routes live in urls.py — completely separate from view logic. Cleaner separation but two files to maintain.

Key difference: Spring Boot and Flask co-locate routing with logic (annotation / decorator on the handler). Django separates routing entirely into urls.py. Django's approach is cleaner at scale — you can read all routes in one place — but requires jumping between files.

GET request — fetch by ID

GET /articles/42 → return article JSON or 404.

Spring Boot
@RestController
@RequestMapping("/articles")
public class ArticleController {

  @Autowired
  private ArticleRepository repo;

  @GetMapping("/{id}")
  public ResponseEntity<Article>
      get(@PathVariable Long id) {

    return repo.findById(id)
      .map(ResponseEntity::ok)
      .orElse(ResponseEntity
        .notFound().build());
  }
}

@Autowired injects the repo. ResponseEntity wraps the response + status code.

Flask
from flask import jsonify, abort

@app.route(
  '/articles/<int:id>',
  methods=['GET'])
def get_article(id):
    article = Article.query\
      .get(id)

    if not article:
        abort(404)

    return jsonify({
      'id': article.id,
      'title': article.title
    })

jsonify() serializes the dict. abort(404) triggers error. No automatic serialization — you build the dict manually.

Django (DRF)
# serializers.py
class ArticleSerializer(
  serializers.ModelSerializer):
  class Meta:
    model = Article
    fields = ['id','title']

# views.py
class ArticleView(APIView):
  def get(self, request, id):
    try:
      a = Article.objects.get(pk=id)
    except Article.DoesNotExist:
      return Response(
        {'error':'Not found'},
        status=404)
    s = ArticleSerializer(a)
    return Response(s.data)

Serializer handles model→JSON. APIView gives you explicit get/post methods per HTTP verb.

Key difference: Spring Boot auto-serializes via Jackson. Flask requires manual dict building. Django/DRF uses an explicit Serializer class — more code upfront but it also handles validation on POST. DRF's Serializer = Jackson + @Valid + DTO all in one.

POST request — create a resource

POST /articles/ with JSON body → validate → save → return 201.

Spring Boot
@PostMapping("/")
public ResponseEntity<Article>
  create(
  @Valid @RequestBody
  ArticleDTO dto) {

  Article a = new Article();
  a.setTitle(dto.getTitle());
  repo.save(a);

  return ResponseEntity
    .status(201)
    .body(a);
}

// DTO with validation
public class ArticleDTO {
  @NotBlank
  private String title;
}

@RequestBody deserializes JSON. @Valid triggers bean validation. Separate DTO class for input.

Flask
from flask import request, jsonify

@app.route('/articles/',
           methods=['POST'])
def create_article():
  data = request.get_json()

  if not data.get('title'):
    return jsonify(
      {'error':'title required'}
    ), 400

  article = Article(
    title=data['title'])
  db.session.add(article)
  db.session.commit()

  return jsonify({
    'id': article.id,
    'title': article.title
  }), 201

request.get_json() parses body. Validation is manual — you check fields yourself. No automatic validation framework.

Django (DRF)
class ArticleView(APIView):

  def post(self, request):
    serializer = ArticleSerializer(
      data=request.data)

    if not serializer.is_valid():
      return Response(
        serializer.errors,
        status=400)

    serializer.save()
    return Response(
      serializer.data,
      status=201)

Serializer validates AND saves. is_valid() checks all field rules. serializer.errors returns structured error map automatically.

Key difference: Flask validation is fully manual — you write every check. Spring Boot uses bean validation annotations (@NotBlank etc). DRF's serializer does validation + deserialization + saving in one object — the most elegant of the three for CRUD APIs once you learn the pattern.

Database / ORM

How models are defined and queried.

Spring Boot (JPA)
// Entity
@Entity
public class Article {
  @Id
  @GeneratedValue
  private Long id;
  private String title;
}

// Repository
public interface ArticleRepo
  extends JpaRepository
    <Article, Long> {}

// Query
repo.findById(id);
repo.findByTitle("hello");
repo.save(article);

Annotations on class. Repository interface auto-generates queries from method names. Hibernate under the hood.

Flask (SQLAlchemy)
# Model
class Article(db.Model):
  id = db.Column(
    db.Integer, primary_key=True)
  title = db.Column(db.String(200))

# Query
Article.query.get(id)
Article.query.filter_by(
  title='hello').first()
db.session.add(article)
db.session.commit()

SQLAlchemy is a separate library you add. Similar pattern to JPA but Python-style. Session-based writes.

Django ORM
# Model
class Article(models.Model):
  title = models.CharField(
    max_length=200)

# Query (QuerySet API)
Article.objects.get(pk=id)
Article.objects.filter(
  title='hello').first()
Article.objects.all()\
  .order_by('-created_at')[:10]

# Save
article.save()
Article.objects.create(
  title='hello')

ORM built-in, no extra library. QuerySets are lazy and chainable. Migrations auto-generated with makemigrations.

Key difference: Django ORM is built-in and tightly integrated — models, migrations, admin all work together automatically. Flask uses SQLAlchemy (separate library, more powerful but more setup). Spring Boot uses JPA/Hibernate via annotations. Django's QuerySet API is the most readable of the three for common queries.

Auth & middleware

How you protect routes and intercept requests.

Spring Boot
// Spring Security config
@Configuration
public class SecurityConfig {
  @Bean
  SecurityFilterChain chain(
    HttpSecurity http) throws Exception {
    http
      .authorizeRequests()
      .antMatchers("/admin/**")
      .hasRole("ADMIN")
      .anyRequest().authenticated();
    return http.build();
  }
}

// Protect a route
@PreAuthorize("hasRole('ADMIN')")
@GetMapping("/admin/data")
public String adminOnly() {...}

Spring Security is a separate module. Powerful but notoriously complex to configure initially.

Flask
# Using flask-login or JWT manually
from functools import wraps

def login_required(f):
  @wraps(f)
  def decorated(*args, **kwargs):
    if not current_user:
      return jsonify(
        {'error':'Unauthorized'}
      ), 401
    return f(*args, **kwargs)
  return decorated

@app.route('/protected')
@login_required
def protected():
  return jsonify({'data': '...'})

No built-in auth. You use decorators (like Python's version of middleware). flask-login or flask-jwt-extended are common add-ons.

Django
# Built-in auth — just use it
from rest_framework.permissions\
  import IsAuthenticated

class ArticleView(APIView):
  permission_classes = [
    IsAuthenticated]

  def get(self, request):
    # request.user is available
    return Response({...})

# settings.py middleware
MIDDLEWARE = [
  'django.middleware.security...',
  'django.contrib.sessions...',
  'django.contrib.auth...',
  'myapp.middleware.CustomMW',
]

Auth is built-in. DRF adds permission_classes for API protection. Middleware is a simple list in settings.py.

Key difference: Django ships with a full user/auth/session system — biggest advantage over Flask. Flask requires assembling auth from libraries. Spring Security is the most powerful but also the most complex to configure. Django hits the sweet spot for most use cases.

Error handling

How each framework handles exceptions and returns error responses.

Spring Boot
// Global exception handler
@RestControllerAdvice
public class GlobalHandler {

  @ExceptionHandler(
    ResourceNotFoundException.class)
  @ResponseStatus(HttpStatus.NOT_FOUND)
  public ErrorResponse handleNotFound(
    ResourceNotFoundException ex) {
    return new ErrorResponse(
      ex.getMessage());
  }
}

// Custom exception
public class ResourceNotFoundException
  extends RuntimeException {...}

@RestControllerAdvice is a global handler. Clean separation — controllers throw, handler catches.

Flask
# Register error handlers
@app.errorhandler(404)
def not_found(e):
    return jsonify(
      {'error': 'Not found'}
    ), 404

@app.errorhandler(500)
def server_error(e):
    return jsonify(
      {'error': 'Server error'}
    ), 500

# In a route
abort(404)  # triggers handler

Simple decorator-based error handlers. abort() triggers them. Minimal but effective for small APIs.

Django (DRF)
# DRF handles common errors
# automatically (404, 405, etc.)

# Custom exception handler
# settings.py
REST_FRAMEWORK = {
  'EXCEPTION_HANDLER':
    'myapp.utils.custom_handler'
}

# utils.py
def custom_handler(exc, context):
  response = exception_handler(
    exc, context)
  if response is not None:
    response.data['status'] = \
      response.status_code
  return response

DRF has sensible defaults — 404, 405, auth errors all handled automatically. Custom handler for global overrides.

Key difference: Spring Boot's @RestControllerAdvice is the most structured — great for large teams. Flask's error handlers are simple but you wire everything manually. DRF gives you the most out of the box — common errors just work, and the custom handler is a clean single override point.

Project structure

How a typical production project is laid out.

Spring Boot
src/
  main/java/com/app/
    controller/
      ArticleController.java
    service/
      ArticleService.java
    repository/
      ArticleRepository.java
    model/
      Article.java
    dto/
      ArticleDTO.java
  resources/
    application.properties
pom.xml

Layered by type (controller / service / repo). Verbose but predictable. Every layer is its own class.

Flask
app/
  __init__.py    # app factory
  models.py      # DB models
  routes/
    articles.py  # Blueprint
    users.py
  services/
    article_svc.py
  schemas/
    article.py   # marshmallow
config.py
run.py

No enforced structure — this is a common convention. Blueprints group related routes. You decide the architecture.

Django
mysite/           # project root
  settings.py
  urls.py         # root router
  wsgi.py
articles/         # app module
  models.py
  views.py
  serializers.py
  urls.py         # app routes
  admin.py        # admin config
  migrations/     # auto-generated
users/            # another app
  models.py
  ...
manage.py

Django enforces app-based modularity. Each app is self-contained. migrations/ is auto-generated — never edit manually.

Key difference: Spring Boot organizes by layer (all controllers together, all services together). Django organizes by feature/app (all article code together). Django's approach scales better for large teams — each feature is a self-contained module. Flask gives you total freedom, for better or worse.
